# REQ: 日報・週報フォーマット更新要件

## 1. 背景と目的
「Tactical Command Center (TCC)」の日報および週報生成機能をより実用的なものにアップデートする。
現在の自動生成ロジックを拡張し、プロジェクト全体の進捗可視化、タスクへのコメント記入の容易化、および翌日/翌週の予定の精度向上を図る。

## 2. 機能要件

### 2.1. プロジェクト進捗率の表示 (GitHub Projects 連携) 【日報・週報 共通】
GitHub Projects V2 と連携し、以下の情報を報告書の冒頭に追加する。

- **抽出対象**:
  - `Status` が `In Progress` または `Todo` のアイテム。
  - 要件：「開始日」と「終了日」が設定されている「進捗管理用 Issue」を優先的に抽出する。
- **表示内容**:
  - プロジェクトごとにグルーピング。
  - 各 Issue について、`タイトル + 進捗状況 (完了サブ課題数 / 全サブ課題数) + 進捗率 (%)` を表示。
  - サブ課題がない場合は「Sub-Issueなし」と表示。
  - ステータスごとに `[In Progress 進捗状況]` と `[ToDo リスト]` で分類。

### 2.2. タスクコメント用の空白と記号の自動挿入 【日報・週報 共通】
「本日/今週の実績」および「明日/来週の予定」にリストアップされる各タスク項目に対し、手動コメントを追記しやすくするためのフォーマット変更を行う。

- **自動挿入内容**:
  - 各タスク行の直後に `\n  ⇒　` (インデント + 矢印 + 全角空白) を挿入。
  - これにより、生成直後にエディタ上でそのままコメントを追記可能にする。

### 2.3. 明日/来週の予定の精緻化 【日報・週報 共通】
現在は単純にアクティブタスクを5件リストアップしているが、これを改善する。

- **表示対象**:
  - GitHub Projects の `Todo` ステータスのアイテム、またはローカルタスクの `todo` ステータスのアイテム。
- **ソート/フィルタ**:
  - 優先度 (`priority`) が高いもの、または期限 (`due_date`) が近いものを優先。

### 2.4. 週報の期間指定 UI 【週報のみ】
「+ WEEKLY」ボタン押下後、即座に生成を開始するのではなく、以下の手順を踏む。

- **フロー**:
  1. ユーザーが `+ WEEKLY` をクリック。
  2. 開始日と終了日を選択するモーダル/ダイアログが表示される。
  3. デフォルト値: 当週の月曜日〜金曜日。
  4. ユーザーが期間を確定後、週報生成が開始される。
- **UI 要件**:
  - 日付ピッカーは既存の `flatpickr` ライブラリを活用する（`package.json` に依存済み）。
  - TCC のサイバー UI テーマに合致したスタイリング。

### 2.5. 週報フォーマットの統一 【週報】
週報のテンプレートを日報の改善内容に準拠させる。

- **セクション構成**:
  1. プロジェクト進捗状況（2.1 と同じ）
  2. 今週の実績（GitHub Activity + Local Tasks + コメント記号）
  3. 来週の予定（ソート改善 + コメント記号）
  4. 特記事項 / 定例メモ（既存）

## 3. 非機能要件

### 3.1. GitHub 非連携時の動作保証（グレースフルデグラデーション）
GitHub PAT が未設定、またはリポジトリが登録されていない場合でも、日報・週報の生成が正常に動作すること。

- **GitHub Activity セクション**: 「（GitHub 連携が設定されていません）」と表示。
- **プロジェクト進捗セクション**: セクションごとスキップ（表示しない）。
- **ローカルタスク**: 正常に取得・表示される。
- **エラーハンドリング**: GitHub API 呼び出しの失敗は握りつぶし、ローカルタスクのみで報告書を生成する。

### 3.2. パフォーマンス
- GitHub API の呼び出し回数を最小限に抑える（キャッシュの活用）。

### 3.3. 可用性
- サブ課題（Sub-issues）機能が利用できない、あるいは PAT に権限がない場合でも、エラーで停止せず「Sub-Issueなし」として処理を続行する。

## 4. UI への影響
- `pages/report.vue` の `+ WEEKLY` ボタン動作の変更（期間選択ダイアログの追加）。
- 生成時プレビューに上記改善を反映した Markdown が表示される。
