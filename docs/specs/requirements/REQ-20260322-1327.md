# REQ-20260322-1327: 統合ダッシュボード機能 (Integrated Mission Briefing)

## 1. 概要

Progress Memo App を「スタンドアロンのタスク管理ツール」から「統合ダッシュボード型のオーケストレーションプラットフォーム」へ進化させる。GitHub Projects / Issues の情報をアプリ内に集約し、ローカルタスクとの一元管理を実現する。あわせて、日報エディタの改善、アキュムレート機能の不具合修正、プロジェクト階層構造の導入を行う。

## 2. 背景

### 2.1. 課題

- **二重管理問題**: Progress Memo App とGitHub（Issues / Projects）の両方に進捗を記載する必要があり、運用負荷が高い。
- **日報エディタの使いにくさ**: 現在のフォーム構造化型エディタは、Google Chat のマークダウン仕様との不整合があり、装飾が意図通りに反映されない。
- **アキュムレート機能の欠損**: existence しないタスクへの蓄積データが破棄される不具合がある。
- **プロジェクト階層の不在**: 複数リポジトリにまたがる案件（プロダクト）を包括的にグルーピングする手段がない。

### 2.2. 目的

- GitHub をデータソースの一つとして取り込み、**書く場所は分散のまま、見る場所を一箇所に集約**する。
- 日報作成のワークフローを改善し、Google Chat への送信精度を向上させる。
- アプリ全体のデータモデルをプロダクト/プロジェクト/タスクの関係で整理する。
- 将来的な AI アシスタント連携（優先度推奨）や Google Calendar 連携の基盤を設ける。

## 3. 機能要件

### FR-001: 統合ダッシュボード画面 (MISSION BRIEFING)

- 説明: アプリ起動時に表示される統合ビュー画面を新設する。以下の情報を一画面で俯瞰できること。
- 表示データ:
  - ローカルの「In Progress」タスク一覧
  - GitHub Projects V2 のステータス別アイテム一覧（Todo / In Progress）
  - 遅延タスクのハイライト表示
- データソース: ローカルMarkdownファイル + GitHub GraphQL API (`viewer.projectsV2`)
- 受入条件:
  - [ ] PAT が設定済みの場合、ユーザーが関与する全 GitHub Projects のアイテムを横断取得できること
  - [ ] ローカルタスクと GitHub タスクが統合された一覧で表示されること
  - [ ] 遅延タスク（期限超過）と期限接近タスク（3日以内）が視覚的に識別できること
  - [ ] PAT 未設定時は GitHub セクションがスキップされ、ローカルタスクのみ表示されること

### FR-002: ガントチャート表示 (OPERATION TIMELINE)

- 説明: GitHub Projects V2 のアイテムを時間軸ベースのガントチャートで可視化する。
- 表示データ:
  - 横軸: 時間（日単位）、今日ラインを赤色で表示
  - 縦軸: プロジェクト別にグルーピングされた Issue 一覧
  - バーの色: 予定通り(青/緑) / 期限接近(黄) / 期限超過(赤) / 完了(緑+消灯)
- データソース: GitHub Projects V2 GraphQL API (Date フィールド / Iteration)
- 受入条件:
  - [ ] GitHub Projects V2 において、開始日・終了日（Date型カスタムフィールドまたはIteration）が設定されているアイテムがバー表示されること
  - [ ] 今日の日付を示す縦ラインが常時表示されること
  - [ ] 表示範囲を月単位（または週単位）でスクロールできること
  - [ ] プロジェクトタグでフィルタリングできること（FR-006 と連携）
  - [ ] 日付フィールドが未設定のアイテムは「日程未定」セクションにリスト表示されること

### FR-003: 日報マークダウンエディタ (REPORT画面 改修)

- 説明: 現在のフォーム構造化型エディタを廃止し、既存の TASKS エディタと同等のマークダウンエディタに置き換える。
- 入力方式:
  - 左カラム: マークダウンテキストエディタ（ツールバー付き）
  - 右カラム: リアルタイムプレビュー（Google Chat 互換モード付き）
- コンテキストメニュー:
  - 「プロジェクトを挿入」: 登録済みプロジェクト名をサブメニューから選択 → `*【ProjectName】*` 形式で挿入
  - 「タスクを挿入」: In Progress タスクをサブメニューから選択 → `　・TaskTitle` 形式で挿入（カーソルは次行の詳細入力位置へ移動）
  - 「テンプレートを挿入」: 日報 / 週報テンプレートの挿入
- Google Chat 互換:
  - プレビューに「Google Chat モード」トグルを設け、Google Chat で有効なマークダウン記法のみでレンダリングする
  - 送信時に標準 Markdown → Google Chat Markdown への自動変換を行う（`**bold**` → `*bold*` 等）
- 受入条件:
  - [ ] マークダウンエディタで自由記述した日報テキストを Google Chat Webhook で送信できること
  - [ ] 右クリックメニューからプロジェクト名・タスク名を挿入できること
  - [ ] Google Chat モードプレビューで、送信後の表示結果を事前確認できること
  - [ ] 既存のアキュムレーション機能（蓄積ログへの反映）がマークダウンベースで動作すること

### FR-004: GitHub Activity 日報統合

- 説明: REPORT画面に「GitHub Activity 取得」機能を追加し、当日のGitHubアクティビティ（コミット、Issue更新、PR）を日報に自動取り込む。
- 取得データ:
  - 当日の Events API (`GET /users/{username}/events`) または Search API
  - コミットメッセージ、Issue への コメント、PR のオープン/マージ
- 出力: エディタのカーソル位置にマークダウンテキストとして挿入
- 受入条件:
  - [ ] 「FETCH GITHUB ACTIVITY」ボタン押下で当日のアクティビティが取得されること
  - [ ] 取得結果がマークダウン形式でエディタに挿入されること
  - [ ] 日付範囲を指定できること（デフォルトは当日）

### FR-005: アキュムレーション未存在タスク自動作成

- 説明: `distribute-accumulation` 処理において、対象タスクファイルが存在しない場合に新規タスクファイルを自動生成する。
- 振る舞い:
  - accumulation.md のメタデータ（project名, title, file名）から frontmatter を構成
  - 対象プロジェクトが `projects.json` に存在しない場合は、プロジェクトも自動追加
  - 生成されたタスクのステータスは `inProgress` とする
- 受入条件:
  - [ ] 存在しないタスクファイルへのアキュムレートが破棄されずに新規ファイルとして保存されること
  - [ ] 新規作成されたタスクが正しいプロジェクトに紐付いていること
  - [ ] 既存タスクへのアキュムレートは従来通り追記されること（デグレードなし）

### FR-006: プロジェクト階層化（タグ方式）

- 説明: `projects.json` のプロジェクト定義に `tags` 属性を追加し、案件（プロダクト）単位の上位グルーピングを実現する。
- データモデル:
  ```json
  {
    "project": {
      "1": { "name": "Inbox (未分類)", "tags": [] },
      "2": { "name": "kiosk-frontend", "tags": ["キオスク端末"] },
      "3": { "name": "kiosk-backend",  "tags": ["キオスク端末"] }
    }
  }
  ```
- UI影響:
  - TASKS画面: タグによるグルーピング表示を追加（トグル式）
  - PROJECTS画面: プロジェクト編集時にタグの追加・編集が可能
  - ガントチャート: タグによるフィルタリング
- 受入条件:
  - [ ] プロジェクトにタグを付与できること
  - [ ] タグでグルーピング/フィルタリングした表示ができること
  - [ ] タスクファイルの frontmatter に変更が不要であること（タグはプロジェクトレベル属性）
  - [ ] タグ未設定のプロジェクトは従来通り動作すること（後方互換性）

### FR-007: GitHub Issue 自動コメント (日報連動)

- 説明: REPORT画面からの日報送信時に、各タスクに紐づく GitHub Issue に対して進捗コメントを自動投稿する。
- 振る舞い:
  - 日報送信成功後、タスクの frontmatter に `issue` (番号) と `issue_url` が存在するものを対象
  - `POST /repos/{owner}/{repo}/issues/{issue_number}/comments` で日報テキスト（当該タスクの部分）を投稿
  - ユーザーに投稿成否をフィードバック
- 受入条件:
  - [ ] 日報送信時に、関連 Issue へ自動的にコメントが投稿されること
  - [ ] Issue 情報が未設定のタスクはスキップされること
  - [ ] コメント投稿の有効/無効を設定で切り替え可能であること

### FR-008: TASK.md への関連 Issue フィールド追加

- 説明: AIエージェント連携のため、TASK.md テンプレート（specification-standards スキル）の「Git運用」セクションに `関連 Issue` フィールドを追加する。
- 変更内容:
  ```markdown
  ## Git 運用 (Source Control)
  - 関連 Issue: #N (repository: owner/repo)
  - ブランチ名: `feature/{N}-{feature-name}`
  - コミットメッセージ: `{type}: {description} #{N}`
  ```
- ワークフロー影響:
  - `commit_save.md`: コミットメッセージ末尾に Issue 番号 `#N` が含まれていることを確認する指示を追加
  - `branch_create.md`: ブランチ名に Issue 番号を含める推奨を追加
- 受入条件:
  - [ ] TASK.md テンプレートに関連 Issue フィールドが追加されていること
  - [ ] commit_save / branch_create ワークフローが Issue 番号を自動参照すること

## 4. 非機能要件

### NFR-001: パフォーマンス

- GitHub GraphQL API 呼び出しは非同期で行い、ローカルデータの表示を阻害しないこと
- ガントチャートの描画は 100 アイテム以下で 1 秒以内にレンダリングされること
- API レスポンスはセッション内でキャッシュし、不要な再取得を避けること

### NFR-002: PAT スコープ

- GitHub Projects V2 へのアクセスには PAT に `read:project` スコープが必要
- 設定画面で PAT のスコープ不足を検知した場合、ユーザーにガイダンスを表示すること
- Issue コメントの投稿には `repo` スコープが必要

### NFR-003: UI 一貫性

- 新規画面（BRIEFING, TIMELINE）は既存の Ace Combat テーマ（ダークブルー基調、フォントサイズ可変、`glass-panel`、`ace-highlight` 等）に準拠すること
- エースコンバットの戦術データリンク UI の世界観を維持すること

### NFR-004: 後方互換性

- 既存のタスク Markdown ファイル（frontmatter 形式）に破壊的変更を加えないこと
- `projects.json` の既存フォーマット（`tags` 未設定）でも正常動作すること
- 既存の TASKS / REPORT / PROJECTS / SETTINGS 画面の基本機能にデグレードが発生しないこと

### NFR-005: オフライン動作

- GitHub API が到達不能な場合でも、ローカルタスク管理機能（TASKS / REPORT / PROJECTS）は正常に動作すること
- API エラー時は明確なメッセージを表示し、ローカルデータのみで動作を継続すること

## 5. 制約事項

- **技術スタック**: Electron (v34) + Nuxt 3 + Vue 3 + Tailwind CSS の既存構成を維持する
- **GitHub API**: GraphQL API (Projects V2) と REST API (Issues, Events) の併用。認証は既存の PAT メカニズムを流用する
- **ローカルデータ形式**: Markdown (gray-matter) + JSON を維持する。新たなデータベースは導入しない
- **Google Chat**: 独自のマークダウンパーサー仕様に準拠する必要がある（`*太字*`, `_斜体_` 等、標準 Markdown との差異あり）

## 6. 実装フェーズ（推奨）

大規模な機能追加のため、段階的な実装を推奨する。

| フェーズ | 内容 | 対象FR |
|---------|------|--------|
| **Phase 1a** | プロジェクトタグ化 + アキュムレート修正 + TASK.mdテンプレート改修 | FR-005, FR-006, FR-008 |
| **Phase 1b** | 日報エディタ改修 + Google Chat 互換 + Issue 自動コメント | FR-003, FR-007 |
| **Phase 2** | 統合ダッシュボード + GitHub Activity 統合 | FR-001, FR-004 |
| **Phase 3** | ガントチャート (OPERATION TIMELINE) | FR-002 |

> Phase 1a はデータモデルとインフラの改善であり、後続の全フェーズの基盤となる。
> Phase 1b は既存画面の改修であり、ユーザー体験の即時改善が見込める。
> Phase 2-3 は新規画面の追加であり、GitHub API 連携を段階的に拡張する。

## 7. 用語定義

| 用語 | 定義 |
|------|------|
| タスク | ローカルの `.md` ファイル 1 つに対応する作業項目 |
| プロジェクト | タスクを束ねるグルーピング単位。`projects.json` で管理。GitHubリポジトリと概ね 1:1 対応 |
| タグ | プロジェクトに付与する上位分類ラベル。案件やプロダクト単位のグルーピングに使用 |
| アキュムレーション | 日報送信時に蓄積される進捗ログ (`accumulation.md`) |
| BRIEFING | 統合ダッシュボード画面の呼称 (MISSION BRIEFING) |
| TIMELINE | ガントチャート画面の呼称 (OPERATION TIMELINE) |
